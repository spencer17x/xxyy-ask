# XXYY 产品客服 RAG 设计

## 背景

当前仓库已经沉淀了 XXYY 产品知识库种子，包括官网清洗文档、页面级 Markdown、页面元数据，以及官方 X 历史推文产品更新汇总。第一期机器人目标是做产品客服问答，不接个人账户、交易查询、链上取证或 MEV 检测工具。

## 目标

- 回答 XXYY 产品功能、使用步骤、权限权益、产品更新等问题。
- 支持中文自然语言提问。
- 回答时引用知识库来源，便于用户追溯。
- 对知识库没有依据的问题明确说明不确定。
- 对个人交易、余额、订单、MEV 检测等实时查询请求做边界识别，并引导用户提供给人工客服或后续工具。
- 为第二期工具调用预留清晰扩展点。

## 非目标

- 不查询用户个人账户、钱包、订单、返佣、余额或交易记录。
- 不判断交易是否被 MEV 或夹子攻击。
- 不做投资建议、喊单建议、买卖点判断或项目推荐。
- 不自动执行任何交易、挂单、监控配置变更或钱包操作。
- 不承诺功能的实时可用状态；当官网文档和社交更新冲突时，以官网文档为准。

## 推荐方案

第一期采用 RAG-first 架构，并保留一个轻量问题分类器。分类器只决定走产品知识问答，还是走超范围响应，不做复杂 agent 循环。

备选方案一是纯 RAG，不做分类器。实现最快，但面对“帮我查交易”“我是不是被夹了”等问题时容易给出泛化回答，边界不够稳。

备选方案二是完整 Agentic RAG。它可以为后续工具调用做好准备，但第一期没有工具，过早引入 agent loop 会增加复杂度和误判风险。

推荐选择轻量分类器加 RAG：它能控制风险，又不会给第一期增加太多工程负担。

## 架构

```text
用户问题
  ↓
输入清洗与语言检测
  ↓
问题分类器
  ├─ 产品知识问题 → 检索 → 重排 → 生成回答 → 来源引用
  ├─ 操作步骤问题 → 检索 → 步骤化回答 → 来源引用
  ├─ 超范围实时查询 → 边界说明 → 人工/后续工具引导
  └─ 投资/喊单问题 → 安全拒答 → 可替代的产品功能说明
```

### 问题分类器

分类器输出固定标签：

- `product_qa`：产品功能、概念、权益、更新内容。
- `how_to`：具体操作步骤，例如如何设置钱包监控或 Telegram 通知。
- `realtime_account_query`：需要查账户、订单、返佣、余额、钱包、交易记录。
- `mev_or_chain_forensics`：询问某笔交易是否被夹、滑点损失、链上交易取证。
- `investment_advice`：买卖建议、项目推荐、喊单、收益承诺。
- `unknown`：意图不清，需要追问。

分类器可以先用 LLM prompt 实现，后续根据日志沉淀规则或小模型。

### 检索层

知识库来源：

- `docs/product-features/xxyy-product-functions.md`
- `docs/product-features/pages/*.md`
- `docs/product-features/manifest.jsonl`
- `docs/product-features/xxyy-x-updates.md`

每个 chunk 应带 metadata：

```ts
{
  id: string;
  title: string;
  module: string;
  sourceType: "official_docs" | "x_updates";
  sourceUrl?: string;
  file: string;
  order?: number;
  retrievedAt?: string;
  content: string;
}
```

推荐使用混合检索：

- 向量检索：处理口语化、同义词和跨语言表达。
- BM25/关键词检索：处理功能名、链名、专有词、文件标题。
- 重排：优先官方文档，其次 X 更新；相同主题优先更具体页面。

### 生成层

回答约束：

- 只基于检索片段回答。
- 回答中保留“来源”列表，至少列出文档标题或文件路径；有 URL 时展示 URL。
- 如果官网文档与 X 更新不一致，以官网文档为准，并说明 X 更新仅作为补充线索。
- 找不到依据时，不要编造。回答“当前知识库没有明确说明”。
- 对操作步骤使用编号列表，避免一次性给过多分支。
- 对风险类问题给产品功能解释，不给投资判断。

### 超范围处理

个人交易或 MEV 相关问题不进入普通 RAG 回答，而是返回边界说明。

示例：

> 当前产品客服机器人暂不支持查询个人交易记录，也不能直接判断交易是否被 MEV 或夹子攻击。判断这类问题需要读取交易 hash、区块/slot、前后交易和池子价格变化。你可以把交易 hash 提供给人工客服，或等待后续链上检测工具上线。

账户、返佣或订单问题示例：

> 当前机器人不能读取你的账户数据。你可以在 XXYY 页面查看对应模块，或联系人工客服处理。下面是相关功能说明：...

投资建议问题示例：

> 我不能提供买卖建议或收益承诺。不过可以介绍 XXYY 中用于观察项目的功能，例如趋势、扫链、钱包监控、Holder 分析和 KOL 买入列表。

## 数据流

1. 定期读取知识库 Markdown 和 manifest。
2. 将文档按标题层级、页面边界和语义段落切分。
3. 为 chunk 写入 metadata，生成 embedding，并建立 BM25 索引。
4. 用户提问进入分类器。
5. 产品知识和操作步骤问题进入检索。
6. 检索结果经过重排与去重。
7. 生成器根据片段回答，并附来源。
8. 记录查询、命中片段、回答和用户反馈，用于后续评测。

## 评测

第一期需要建立一个小型 golden set，覆盖常见客服问题：

- 钱包监控如何设置 Telegram 通知？
- Pro 和永久 Pro 有什么区别？
- 扫链页面可以筛选哪些条件？
- 自动止盈止损怎么理解？
- BSC/Base/ETH 支持哪些功能？
- X 更新里提到的 API Skill 是什么？
- 如何处理“帮我看交易是否被夹了”？
- 如何处理“帮我推荐今天买哪个币”？

每条样例评估：

- 是否命中正确来源。
- 是否回答完整。
- 是否避免编造。
- 是否正确识别超范围。
- 是否给出合适的下一步引导。

## 错误处理

- 检索无结果：说明知识库没有明确答案，并建议用户联系人工或补充问题。
- 检索结果低置信：输出保守答案，并标注“根据目前检索到的资料”。
- 多个来源冲突：优先官方文档，并说明社交媒体更新可能是历史版本或补充信息。
- 用户问题含交易 hash：识别为后续工具场景，不解析、不推断。
- 用户要求交易操作：拒绝执行，改为说明相关功能入口或操作风险。

## 后续扩展

第二期可以在分类器后增加工具层：

- 查询订单状态。
- 查询返佣或 Cashback。
- 查询钱包监控配置。
- 查询用户交易记录。
- 判断交易是否疑似被 MEV 夹。

这些工具必须返回结构化证据，LLM 只负责解释结果，不能自行做链上结论。

## 成功标准

- 产品功能类问题能稳定引用正确文档。
- 操作步骤类问题能输出清晰步骤。
- 超范围问题不会误导用户以为机器人已经查过实时数据。
- 投资建议类问题不会给出买卖判断。
- 后续可以不重写主流程，直接在分类器后接入工具调用。
